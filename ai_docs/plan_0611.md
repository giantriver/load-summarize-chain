# PDF / 文字切分流程改良計畫（修訂版）

請協助我修改目前的 PDF / 文字切分流程，目標是加入「自動文件結構偵測」與「章節階層 metadata 保留」。

目前系統已有 PDF 文字擷取、文字清理、結構切分、fallback token splitter、短 chunk 修補等流程。請在既有架構上改良，不要整個重寫。

> 本版已根據第一次審查意見修訂。修訂重點集中在五個「必須調整」項目與若干非阻斷建議，詳見最後「附錄：審查意見對應表」。

---

## 目標

實作一套統一的文件結構判定流程：

1. 所有 heading regex 同時偵測（不預先選一套）。
2. 每一行先轉成 heading candidate，並保留行級來源座標 `(page, line_no, char_start, char_end)`。
3. 過濾頁碼、目錄、附圖/附表目錄、圖表標題、表格資料等雜訊。
4. 統計整份文件實際出現的 heading pattern（structure profile）。
5. 由統計結果「資料驅動」推斷每種 heading style 的 level（不寫死）。
6. 用 stack 建立文件結構樹。
7. 切 chunk 時，以「通過過濾的 candidate 行位置」為邊界，每個 chunk 保留 `source_file`、`page_start`、`page_end`、`heading_path`、`breadcrumb` 等 metadata。

## 主要支援文件格式

### A. 技術報告 / 計畫書格式（例：ZPRL / TRR 除役計畫書）

```text
1. 綜合概述
1.1 概論
1.1.1 緣由及目的
(1).緣由
(A).預備措施
(a)依據各輻射作業特性...
```

也要支援沒有空白或沒有點號的情況：

```text
1 綜合概述
1.1概論
1.1.1緣由及目的
10.3.5合理抑低措施
```

### B. 中文法規 / 作業要點格式（例：境外核災處理作業要點）

```text
一、目的
二、適用時機
三、作業程序
（一）二級開設
（二）一級開設
1. 開設時機
2. 參與機關
（1）境外部分
（2）邊境部分
```

> 注意：B 格式文件本身就含有 `1. 2. 3.` 這種單層阿拉伯數字條列，且 `（1）（2）` 會掛在 `1.` 之下。level 推斷必須能正確分辨這兩者，見下方「Level 推斷（資料驅動）」。

## Heading pattern（全部一起跑）

```python
HEADING_PATTERNS = [
    # 1 / 1.1 / 1.1.1 / 10.3.5 ；分隔符容許 . ． 、
    ("decimal", re.compile(
        r"^(?P<num>\d+(?:[.．]\d+)*)[.．、]?\s*(?P<title>[一-鿿A-Za-z].+)$"
    )),

    # (1) (1). （1）
    ("paren_num", re.compile(
        r"^[\(（](?P<num>\d+)[\)）][.．]?\s*(?P<title>.+)$"
    )),

    # (A) (A). （A）
    ("paren_upper", re.compile(
        r"^[\(（](?P<num>[A-Z])[\)）][.．]?\s*(?P<title>.+)$"
    )),

    # (a) （a）
    ("paren_lower", re.compile(
        r"^[\(（](?P<num>[a-z])[\)）][.．]?\s*(?P<title>.+)$"
    )),

    # 一、目的
    ("cjk_comma", re.compile(
        r"^(?P<num>[一二三四五六七八九十百]+)、\s*(?P<title>.+)$"
    )),

    # （一）目的
    ("cjk_paren", re.compile(
        r"^[（(](?P<num>[一二三四五六七八九十百]+)[）)]\s*(?P<title>.+)$"
    )),
]

# bare number：序號與標題被 PDF 拆成兩行，例如獨立一行 "1." 後接 "綜合概述"
BARE_NUMBER_RE = re.compile(r"^(?P<num>\d+(?:[.．]\d+)*)[.．、]$")
```

`detect_heading_candidates` 對每行先嘗試 `BARE_NUMBER_RE`：若命中且下一行為非空、且下一行本身不是 heading，則把兩行合併後再套 `HEADING_PATTERNS`。

## 流程順序（重要修正）

```text
build_line_records(pages)          # 行級結構，保留 (page, line_text)
        ↓
clean_line_records()               # 清頁碼 / 合併空行 / unwrap，逐行做、保留 page 歸屬
        ↓
detect_heading_candidates()        # 產生 candidate，含 (page, line_no)
        ↓
find_toc_regions() / find_body_start_page()
        ↓
filter_toc_and_noise()             # is_noise_line + TOC 區排除
        ↓
analyze_structure_profile()
        ↓
infer_levels()                     # 資料驅動 level mapping
        ↓
validate_sequence()                # 序號連續性檢查，剔除偽 heading
        ↓
build_heading_tree()
        ↓
split_by_candidate_lines()         # 以通過過濾的 candidate 行為切分邊界
        ↓
若區塊超過 chunk_size，依下一層 candidate 拆
        ↓
若仍超過 chunk_size，fallback 到 RecursiveCharacterTextSplitter
        ↓
repair_small_chunks()              # 對 (text, heading_context) 單元操作
        ↓
轉成 LangChain Document，保留 metadata
```

### 修正 1：`unwrap` 必須與新 pattern 同步（必須）

目前 `clean_document_text()` 內含 `unwrap_pdf_line_breaks()`，它透過 `should_keep_line_break()` →
`is_standalone_section_heading()` / `is_heading_line()` 決定要不要保留換行。這兩個判斷目前只認得舊的
5 個 regex，因此新格式（`1.1概論`、半形 `(1).緣由`、`(A).`、`(a)`）會在「偵測之前」就被黏回上一行，
candidate 直接消失。

**修法**：`is_heading_line()` / `is_standalone_section_heading()` 一律改用同一份 `HEADING_PATTERNS`
（含 `BARE_NUMBER_RE`）。`unwrap` 只會「保留換行」不會破壞文字，所以擴大可辨識 heading 是安全的，
只是讓更多正確的 heading 邊界被保留下來。

### 修正 2：清理改成行級、保留 page 來源（必須）

現行做法是「逐頁 `clean_document_text()` → 合併成 `combined_text` → `combined_text.find(chunk)` 回推頁碼」。
`clean` 與 `unwrap` 會改變文字，使「candidate 的 `(page, line_no)`」與「chunk 在 `combined_text` 的字元位置」
分屬兩套座標系，頁碼會對不準。

**修法**：把清理流程改成操作「行紀錄串列」，每筆保留 `page` 來源：

```python
@dataclass
class LineRecord:
    page: int
    text: str          # 清理後文字
    char_start: int    # 在 combined_text 中的字元起點
    char_end: int
```

- `unwrap` 合併兩行時，合併後該行的 `page` 取「前一行的 page」，`char_end` 延伸到後一行尾。
- `combined_text` 由這些 `LineRecord` 串接而成，candidate、chunk、頁碼共用同一座標系。
- chunk 的 `page_start` / `page_end` = 該 chunk 涵蓋的 `LineRecord.page` 之 min / max。

> 這是本計畫工程量最大的一塊（不是重寫，但要把清理改成行級），估時要算進去。

## 雜訊過濾規則

### `is_noise_line(line)`

請避免把以下內容當成正文 heading：

1. 單純頁碼：`1` / `2` / `i` / `ii` / `iii`（整行只有阿拉伯數字或羅馬數字）。
2. 目錄 / 頁次 / 附圖目錄 / 附表目錄標題：`目錄` / `頁次` / `頁 次` / `附圖目錄` / `附表目錄` / `圖 目 錄` / `表 目 錄`。
3. 目錄列（含點引導線 + 行尾頁碼）：

```text
1.1.1 緣由及目的..................................................1
10.3.5 合理抑低(ALARA).........................................164
```

判斷：行內含 3 個以上連續 `.`／`．`／`…`，或「行尾為 `\s*\d+$`」且整行像「標題……頁碼」。

4. 圖表標題（本文中的 caption，半 / 全形冒號、`-` 或 `.` 編號都要涵蓋）：

```text
圖 1-1：ZPRL(004 館)於本院院區位置圖
表 1-1：ZPRL 系統現況及除役範圍
圖 2.1：TRR設施位於本院院區位置圖
表2.1：台灣地震規模大於6災害統計表
```

判斷：`^[圖表]\s*\d+([-.–]\d+)?\s*[：:]`。

5. DOC / PDF 轉文字常見的目錄黏連碼：

```text
24410 輻射劑量評估及輻射防護措施
24410.1民眾劑量評估
28511 環境輻射監測
```

這類請靠「TOC 區排除」+「序號連續性檢查」處理（單一 `is_noise_line` 規則擋不乾淨）。

### TOC 區與正文起點

- `find_toc_regions(pages)`：偵測到 `目錄 / 頁次 / 附圖目錄 / 附表目錄` 標題後，到「第一個真正正文大章」
  （`1. 綜合概述`、`1 綜合概述`、`一、目的`）之間，整段標記為 **TOC 區**。
- `find_body_start_page(pages)`：回傳第一個正文大章所在頁。**B 格式（法規）通常沒有目錄頁，預設回第 1 頁。**

**重要分工（避免誤殺正文）**：

- **內容收錄範圍** 只由「明確偵測到的 TOC 區」決定：TOC 區的文字不進 chunk（對 map-reduce 摘要而言是純雜訊），
  並在 Streamlit preview 顯示「已排除 N 頁目錄」供人工確認。
- **結構偵測起點**（`body_start_page`）只用來決定「從哪裡開始信任 heading 結構」。即使它判斷錯，
  也**不可**因此丟掉正文內容；非 TOC 區的文字一律保留。

## Level 推斷（資料驅動，核心修正）

### `analyze_structure_profile(candidates)`

```python
{
    "style_counts": Counter({...}),
    "decimal_levels": Counter({1: 17, 2: 80, 3: 120}),   # key = num.count('.') + 1
    "max_decimal_depth": 3,        # 只計入 count >= MIN_DEPTH_SUPPORT 的深度，避免雜訊抬高
    "has_decimal": True,           # 見下方定義
    "has_cjk": False,
}
```

### `has_decimal` 的定義（必須修正）

**不可**用「是否出現過 decimal」判斷。B 格式法規本來就有 `1. 2. 3.` 單層數字條列，若用舊規則會把法規文件
誤判成技術報告格式，導致 `1.` 被當 level 1、與 `一、` 同層，整棵樹壞掉。

正確定義：

```python
multi_decimal = sum(c for depth, c in decimal_levels.items() if depth >= 2)
has_decimal = multi_decimal >= MIN_MULTI_DECIMAL   # 例如 3
```

也就是「存在足夠數量的『深度 ≥ 2』小數編號（出現過 1.1 / 1.1.1）」才算技術報告主結構。
法規文件只有單層 `1.`，`has_decimal=False`。

### `infer_levels(candidate, profile)`（資料驅動公式）

不要寫死 4/5/6。level 由 profile 計算，使 paren 區塊永遠緊接在「最深的數字/條號層」之下：

```text
若 has_decimal（技術報告體系）:
    deepest = max_decimal_depth                  # 例如 3
    decimal(depth d)  -> level d                 # 1 / 1.1 / 1.1.1 = L1 / L2 / L3
    cjk_comma         -> level 1                 # 罕見的雜散條號
    cjk_paren         -> level 2

否則（法規體系，has_decimal=False）:
    deepest = 2 + max_decimal_depth              # cjk 佔 1、2；單層 decimal 視為條列
    cjk_comma         -> level 1
    cjk_paren         -> level 2
    decimal(depth d)  -> level (2 + d)           # 單層 1. = L3

兩體系共用（paren 區塊接在 deepest 之下）:
    paren_num   -> level deepest + 1
    paren_upper -> level deepest + 2
    paren_lower -> level deepest + 3
```

**為什麼這樣設計**（同時通過簡化測試與真實文件）：

| 情境 | max_decimal_depth | deepest | `1.`(decimal) | `（1）`(paren_num) |
|---|---|---|---|---|
| 測試 2（純法規，無小數） | 0 | 2 | — | **L3** ✓ 符合測試期待 |
| 真實文件 B（法規含單層 `1.`） | 1 | 3 | **L3** | **L4** ✓ `（1）` 正確掛在 `1.` 下 |
| 技術報告 A（小數到 3 層） | 3 | 3 | L1/L2/L3 | **L4** ✓ 符合測試 1 |

這解決了「測試案例 `（1）=L3`」與「真實文件 `（1）` 應掛在 `1.` 之下（=L4）」的矛盾——
因為 stack 建樹只看相對大小，絕對 level 由 profile 決定，兩者都正確。

> 註：這仍是「單一 profile / 單一 level mapping」，不是切兩套流程。候選偵測、建樹、切分共用同一套程式。

### `validate_sequence(candidates)`（序號連續性檢查，必須新增）

`is_noise_line` + TOC 過濾之後，再做一道序號合理性檢查，這是「目錄過濾之外最有效的單一去噪手段」。
它能擋掉表格數字列與黏連目錄碼，例如：

```text
1904 年 4 月 24 日 嘉義 6.1 3 人死亡     # num=1904，title="年 4 月..."，會被 decimal 誤命中
2.24x10                                  # num=2.24，title="x10..."
50 公噸                                  # num=50，title="公噸"
24410 輻射劑量評估及輻射防護措施          # 黏連目錄碼
```

規則（容許小跳號，例如 ≤ 2）：

- `decimal` level 1（章）：`num` 必須是 `1`，或等於前一個已接受的同層 sibling +1（容許小跳號）。
  → `1904`、`50`、`24410` 因為前面沒有 `1903 / 49 / 24409` 這種章，全部剔除。
- `decimal` 深度 ≥ 2：父層前綴（去掉最後一段）必須是已接受的 heading，或同層存在前驅 sibling。
  → `2.24` 因為沒有已接受的父層 `2` 在該上下文、且最後一段不是 1，被剔除。
- 被剔除者**不進 heading tree**，但其文字仍當正文保留（只是不作為切分邊界）。

## `build_heading_tree(candidates)`

```python
def build_heading_tree(candidates):
    root = {"level": 0, "title": "ROOT", "children": []}
    stack = [root]
    for h in candidates:                     # 已含 level，且已過濾
        level = h["level"]
        while stack and stack[-1]["level"] >= level:
            stack.pop()
        node = {**h, "children": []}
        stack[-1]["children"].append(node)
        stack.append(node)
    return root
```

輸入：

```text
1. 綜合概述 / 1.1 概論 / 1.1.1 緣由及目的 / (1) 緣由 / (2) 目的 / 1.1.2 專有名詞
```

應形成：

```text
1. 綜合概述
└── 1.1 概論
    ├── 1.1.1 緣由及目的
    │   ├── (1) 緣由
    │   └── (2) 目的
    └── 1.1.2 專有名詞
```

## chunk 切分策略

### 修正 3：以 candidate 行位置切分，不可重跑 regex（必須）

目前 `split_by_heading_pattern()` 是拿 regex 對全文「重新 match」。若建完樹後切分時又重跑 regex，
前面做的 TOC / 雜訊 / 序號過濾會全部白費（圖表標題、目錄列會再次變成切分點）。

**修法**：`split_by_candidate_lines()` 一律以「通過過濾後保留下來的 candidate 的 `line_no`」作為切分邊界。
不要再對全文重新比對 pattern。

切分原則維持：**文件結構優先，token 長度保護**

```text
依 heading tree 的最高層 candidate 切區塊
↓ 區塊 > chunk_size
依該區塊內「下一層」candidate 再切
↓ 仍 > chunk_size
fallback 到 RecursiveCharacterTextSplitter.from_tiktoken_encoder()
```

### 修正 4：`repair_small_chunks()` 對 (text, heading_context) 操作（建議）

標題型小 chunk 併入下一塊後，`heading_path` 應反映合併後內容。請把 `repair_small_chunks()` 從「純字串合併」
改成對 `(text, heading_context)` 單元操作；合併後重新由 candidate 位置推導 `heading_path` / `breadcrumb`。
合併方向與「最多 3 輪、合併後不得超過 chunk_size」維持原邏輯。

## 重要限制

1. 不要寫死「一、一定是第一層」或「1. 一定是第一層」；先偵測所有 candidate，再由 profile 決定 level。
2. 第一版不要用目錄建立正文結構；TOC 區只用來「排除」，不建樹。
3. 不要把圖、表標題當章節 heading（之後可另做 figure/table metadata）。
4. 不要破壞既有 fallback：結構切分失敗或區塊過長時，仍 fallback 到
   `RecursiveCharacterTextSplitter.from_tiktoken_encoder()`。
5. 不要讓短標題獨立成無意義 chunk；保留並改良 `repair_small_chunks()`。
6. 已知限制（暫不處理）：整份文件採「單一 profile」。若一份文件前半是法規格式、後半附錄是技術報告格式，
